# Factory: детализированная реализация (стадии 1–2)

## 1) Буферы материалов и «Вариант 2» для складского сырья

### Проблема, которая была
На старте у ранних станков отображались слишком большие входные очереди, потому что клиент сразу суммировал **все будущие потребности** (`consumes`) для ещё не начатых операций.
Это выглядело так, как будто детали заранее «лежат в очереди», хотя фактически они ещё не произведены предыдущими операциями.

### Исправленная модель (client-side, визуальная)

#### 1.1. Что считаем «реальным WIP»
Для каждого заказа идём по операциям в технологической последовательности:

1. Берём операцию `op_i` и её слот расписания `sch_i`.
2. Если `sch_i.end_min > now`, операция ещё не завершена — на буферы она пока не влияет.
3. Если операция завершена (`sch_i.end_min <= now`), смотрим следующую `op_{i+1}`:
   - если следующая есть и **ещё не стартовала** (`sch_{i+1}.start_min > now`), то произведённые `op_i` детали лежат во **входе** следующего станка;
   - если следующей нет (финальная операция) или уже нет куда передавать, детали лежат в **выходе** текущего станка.

Ключевой эффект: во входе станка лежит только то, что уже реально произведено вверх по маршруту, а не «будущая потребность». Поэтому «огромные очереди на старте» исчезают.

#### 1.2. Складское сырьё (внешние материалы) — выбранный Вариант 2
Сырьё не моделируем как полноценные единицы в логике производства, а показываем как визуальный индикатор:

1. Для каждого станка определяем множество **внешних** материалов, которые хоть раз потребляет любая назначенная ему операция.  
   Внешним считаем материал, который либо явно помечен `supplier_*`, либо вообще нигде в рецептах не производится (`not in produces`).
2. Для каждого такого материала в входном буфере отображаем базовую пачку `10`.
3. Если какая-либо операция на этом станке стартовала **менее 2 минут назад**, вычитаем из пачки её расход по этим внешним материалам.
4. По истечении 2 минут вычитание автоматически исчезает (снова показывается 10).

Это даёт нужный «визуальный импульс расхода» без сложной складской модели.

## 2) Логика расчёта игровой ситуации на клиенте

### 2.1. Источник правды в MVP
Сейчас в фронте используется режим `useClientSimulation = true`:

- состояние берётся один раз с сервера (`/api/state`) при `init()`;
- дальше время и простая эволюция состояния идут локально в `simulateTick(...)`;
- сервер используется для действий пользователя (`replan`, `random event`, `reset`, и т.д.).

Это нормально для демо-режима: быстро, стабильно, без частых сетевых зависимостей.

### 2.2. Время и частота обновления (исправлено)
Требование из геймплея:
- 1x: 1 игровая минута/секунда, 1 обновление/сек;
- 4x: 4 обновления/сек;
- 20x: 10 обновлений/сек.

Используем явное соответствие:
- `1x` → каждые `1000ms` + `1` игровая минута;
- `4x` → каждые `250ms` + `1` игровая минута;
- `20x` → каждые `100ms` + `2` игровые минуты.

Почему для 20x по 2 минуты: так соблюдается частота 10 Гц и суммарная скорость 20 игровых минут/секунду.

### 2.3. Планировщик тиков на клиенте
Вместо фиксированного `setInterval(..., 1000)` используется динамический таймер с перезапуском при смене скорости:

1. При смене скорости пересчитываем параметры тика и перезапускаем таймер.
2. Таймер работает через `setInterval` с нужным `intervalMs`.
3. На каждом тике:
   - обновляем `state.now` (или вызываем `/api/tick` в серверном режиме);
   - перерисовываем readout времени и экраны.

Это убирает проблему «время не меняется» при скоростях выше 1x и делает поведение строго по спецификации.

## 3) Дополнительно: кнопки toolbar в module-script

Поскольку `app.js` подключён как `type="module"`, функции не попадают в глобальную область автоматически.
Чтобы inline `onclick="..."` в `index.html` работали стабильно, в `init()` функции явно пробрасываются в `window` (`setSpeed`, `doReplan`, `reset`, модалки и т.д.).

Это предотвращает немые ошибки при кликах и делает управление предсказуемым.

## 4) Синхронизация client-sim с серверными действиями и перегрузка

- Кнопки `Сброс`, `Перепланировать`, `Случайное событие` вызывают серверные endpoint-ы, поэтому в режиме `useClientSimulation=true` клиент обязан подхватывать их ответ и перезаписывать `localState`.
- Если этого не делать, UI выглядит «сломанным»: сервер уже пересчитал состояние, а экран продолжает жить на старом локальном snapshot.
- Для этого после серверных действий делаем `syncLocalState(response_state)` и только потом `refresh()`.

Отдельно по событию `overload`:
- В локальной симуляции событие генерируется случайно так же, как и остальные типы.
- Ранее оно почти не исчезало (было `resolved=false` слишком долго), поэтому «перегрузка на Сборка-1» могла висеть постоянно.
- Добавлен авто-resolve событий по TTL (для `overload` — 20 игровых минут), чтобы индикатор перегрузки не зависал бесконечно.
