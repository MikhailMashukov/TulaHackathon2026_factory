# Весёлый плановик — MVP web-симулятор планирования производства

> Кейс task3 (Райтек ДТГ) на TulaHack 2026.
> Команда: Михаил + Claude Code. Окно: 36 часов.
> Источник ТЗ: `/project/TulaHackathon/docs/tasks/task3_factory.md`.
> Анализ: `/project/TulaHackathon/docs/tasks_analysis/task03_analysis.md`.
> Формат спеки: AgentsOS (`/project/AgentsOS/system/tasks/spec_format/spec_format_spec.md`), вариант #14 — суть как текст под `#`, стадии как `##`, секции как `###`.

Веб-симулятор планирования цеха («Весёлый плановик»): 5+ станков, 3+ типа изделий, диаграмма Ганта, случайные события (поломки/срочные заказы/нет оператора), кнопка перепланирования и LLM-копилот, который в 2 секунды объясняет «перенеси операцию X на станок Y, потеряешь 8 минут вместо 40, дедлайн заказа №3 спасёшь». Всё end-to-end играбельно из браузера: Гант рисуется, события триггерятся, копилот отвечает, баллы считаются.

Зачем: жюри Райтек (Дрокин/Власов — спикеры по APS/RaytecAPS) узнают в этом свой Ai2B. Реалистичный шанс контракта (выручка >1 млрд ₽, AI-направление через Кудряшова, в роадмапе ПРОМОМЕД 2026 «помощник плановика»). Дополнительно — это «своя территория» Михаила: Factorio 650ч + готовый `Factorio_API.md` + `real_production_planning.md` дают доменную глубину, которую большинство веб-команд в 80-командном поле не сделают.

### Минимальная функциональность (буквально из ТЗ)

- ≥5 производственных единиц (станков/участков).
- ≥3 типа изделий с разной последовательностью операций.

### Технические выборы (с обоснованием под Михаила)

- **Backend**: Python 3.12 + FastAPI. Михаил — системщик C++/CUDA/Python, FastAPI родной; в 36 ч нет времени на TypeScript-полиглотство.
- **Frontend**: HTML + vanilla JS + `vis-timeline` через CDN. Михаил не фронтендер — никакого Vue/React, только готовая Гант-библиотека и минимальный обвес. Tailwind CDN для быстрой стилизации.
- **State**: in-memory (dataclass + dict в `app.state`). Никакой БД — 36 часов, single-player, перезагрузка приемлема.
- **Гант-библиотека**: `vis-timeline` (FOSS, MIT, проста, поддерживает группировку по машинам и drag-and-drop из коробки). Альтернативы: `frappe-gantt` (хуже с ресурсами), `dhtmlx-gantt FOSS` (тяжелее).
- **Планировщик**: greedy + EDD (Earliest Due Date). Никакого MILP/CP-SAT — кейс прямо просит «не требуется сложная математическая оптимизация». OR-Tools — overkill и риск утопиться.
- **Время симуляции**: непрерывный ход времени на клиенте с множителем скорости (`1x`, `4x` и т.п.). Backend получает частые малые тики через API.

### Формула баллов (5 компонент)

Все компоненты в [0..100], итоговый score = взвешенное среднее.

| Компонент | Формула | Вес |
|---|---|---|
| `on_time` | (заказов в срок / всего заказов) × 100 | 0.35 |
| `idle` | 100 − (суммарный idle станков / общее время × 100) | 0.20 |
| `balance` | 100 − std(загрузка машин) × 100 (нормированная) | 0.15 |
| `defects` | 100 − (брак-операций / всего операций × 100) | 0.10 |
| `resilience` | 100 − (сорванных дедлайнов после события / событий × 100) | 0.20 |

Итого: `score = sum(weight * component for ...)`. Простая, диффuses (всё и срок, и баланс, и брак), без фантастики.

### API endpoints (FastAPI)

| Метод | Путь | Что делает |
|---|---|---|
| GET | `/api/state` | Полный JSON game state. |
| POST | `/api/tick` | Сдвинуть симуляцию на N минут (вызывается клиентским непрерывным таймером маленькими шагами). |
| POST | `/api/schedule/replan` | Перепланировать с нуля greedy+EDD. |
| POST | `/api/schedule/move` | Body: `{op_id, machine_id, start_min}` — двинуть операцию вручную (drag-and-drop из vis-timeline). |
| POST | `/api/event/random` | Сгенерировать случайное событие. |
| POST | `/api/event/{event_id}/resolve` | Закрыть событие (для поломки — после `duration_min`). |
| POST | `/api/copilot/suggest` | Body: `{event_id?, free_text?}`. Возвращает `{advice_ru, suggested_moves: [...]}` через Claude (или canned fallback). |
| POST | `/api/copilot/apply` | Применить `suggested_moves` атомарно. *(TODO — пока не реализован в коде)* |
| POST | `/api/reset` | Сбросить партию к начальному состоянию. |
| GET | `/` | Статика `index.html`. |

#### Запуск uvicorn вне Docker'а при редактировании из Claude Code внутри Docker'а

Короткий ответ: **в общем случае — нет, `--reload` молча перестаёт срабатывать; надёжно работает только когда uvicorn и редактор кода живут в одном FS-пространстве.** Развёрнуто — три сценария, как это бывает на ноутбуке Михаила (Win10 → WSL2 → Docker с Claude Code, 15 ГБ лимит RAM).

| Сценарий | Где лежат файлы | Где правит Claude Code | Где запущен uvicorn `--reload` | `--reload` подхватит? |
|---|---|---|---|---|
| A. **Всё в Docker** (рекомендуемый) | внутри контейнера (`/project/...`) | внутри контейнера | внутри того же контейнера (`uvicorn src.main:app --reload --host 0.0.0.0 --port 80`), порт проброшен через `-p 6790:80` | Да. Нативный inotify Linux, `watchfiles` ловит все правки мгновенно. |
| B. **Файлы на Win-хосте через bind-mount**, uvicorn на хосте (Windows/PowerShell) | `C:\project\...` (bind-mount в Docker как `/project/...`) | внутри Docker'а через bind-mount | на Win-хосте (`python -m uvicorn ...`) | Частично. Windows `watchfiles` работает через `ReadDirectoryChangesW`, но события от правок, пришедших через Docker VM/WSL2 в bind-mount, часто не пробрасываются или приходят с задержкой. Лечится `--reload-delay 1 --reload-include "*.py"` + запасной polling (`WATCHFILES_FORCE_POLLING=true`), но надёжность так себе. |
| C. **Uvicorn на Win-хосте**, файлы — только внутри Docker | в overlayfs контейнера | внутри Docker'а | на Win-хосте | Нет. Win-процесс физически не видит файлов контейнера — ему нечего перечитывать. Надо сперва bind-mount (и см. сценарий B). |

Вывод для нашего кейса: **вариант A**. На стенде и на ноутбуке Михаила запускаем uvicorn тем же процессом в том же контейнере, где сидит Claude Code (можно в отдельном bash-окне `docker exec -it <container> bash`); порт пробрасываем наружу один раз при старте контейнера (`-p 6790:80`). Браузер (Chrome на Windows) стучится на `localhost:6790` — Docker Desktop сам проксирует. `--reload` видит правки, которые Claude Code делает в `/project/TulaHackathon/factory/src/*.py`, через нативный Linux-inotify.

Доп. практика: в README.md написан запуск `uvicorn src.main:app --reload --host 0.0.0.0 --port 80` (порт 80, согласно стадии 2) — это как раз сценарий A. (NB: README.md пока может содержать старый порт 8000 — при случае обновить.) `--host 0.0.0.0` нужен именно для того, чтобы порт был доступен снаружи контейнера. Если `--reload` всё-таки не подхватывает (в Docker/WSL2 изредка бывает из-за overlayfs inotify-лимитов), запасной вариант — `--reload-delay 0.5` или переменная `WATCHFILES_FORCE_POLLING=true`.

Что сознательно НЕ в requirements (и почему — чтобы не разрастись):

- `sqlalchemy` / `alembic` — состояние in-memory (`app.state`), БД не нужна (стадия 2 → State).
- `numpy` / `pandas` — формула баллов (5 компонент) считается на чистом Python в `src/scoring.py`, объёмы данных тривиальные (десятки заказов, сотни операций).
- `ortools` / `pulp` — кейс прямо не требует MILP/CP-SAT; используем greedy+EDD (стадия 2 → Планировщик). OR-Tools — тяжёлый и риск утопиться.
- `python-multipart` — нет загрузки файлов от пользователя, все запросы JSON.
- `jinja2` — единственный HTML-файл (`index.html`) отдаётся статикой, шаблонизатор не нужен.
- `python-dotenv` — `ANTHROPIC_API_KEY` читаем из `os.environ`, а на стенде ключ задаётся либо в shell, либо есть canned-fallback.
- Frontend-зависимости (vis-timeline, Tailwind) — подключаются с CDN, не через pip/npm; цель — нулевая фронтенд-toolchain (Михаил не фронтендер, стадия 2).

#### FastAPI vs Django — развёрнутое сравнение под наш кейс

| Критерий | FastAPI | Django | Что побеждает для «Весёлого плановика» |
|---|---|---|---|
| **Парадигма** | ASGI, async/await native. | WSGI исторически; Django 4+ умеет ASGI, но полноценный async-стек (ORM, middleware) всё ещё местами синхронный. | FastAPI. Вызов Claude из `/api/copilot/suggest` — это I/O на 1–3 секунды; синхронный Django-воркер заблокирует поток, и параллельный `/api/tick` будет ждать. |
| **Батарейки** | Роутинг + валидация (pydantic) + OpenAPI. Всё остальное — сам. | ORM + migrations + admin + auth + sessions + forms + CSRF + templating. | Django даёт то, что нам **не нужно**: БД нет (in-memory `app.state`), юзер один, форм нет, шаблон один и тот статический. В 36 ч лишние батарейки = лишний шум в голове. |
| **Код endpoint'а** | `@app.post("/api/schedule/move")` + pydantic-модель — 5 строк. | `urls.py` + `views.py` (class/function-based) + `serializers.py` (через DRF) + регистрация приложения в `settings.py` + `AppConfig` — 4 файла, ~30 строк минимум. | FastAPI. 9 endpoint'ов из стадии 2 на Django = ~270 строк boilerplate, на FastAPI = ~60. |
| **Валидация входа** | pydantic встроен, сам даёт 422 с понятной ошибкой. | DRF-serializers (отдельный пакет, своё API) или ручная валидация в view. | FastAPI. Drag-and-drop из vis-timeline может прислать мусор — валидация «из коробки» нужна сразу. |
| **OpenAPI/Swagger** | `/docs` авто-генерируется из сигнатур функций. | Через `drf-spectacular` или `drf-yasg`, настройка. | FastAPI. На хакатоне удобно открыть `/docs` и потыкать endpoint'ы руками без Postman. |
| **Стоимость старта** | `uvicorn src.main:app --reload` — одна команда. | `django-admin startproject` + `startapp` + `settings.py` (INSTALLED_APPS, DATABASES=sqlite? нет БД, но Django ругается) + `migrate` + `runserver`. | FastAPI. У нас 36 часов, не 36 дней. |
| **Тяжесть зависимости** | `fastapi` + `starlette` + `pydantic` + `uvicorn` — ~15 МБ. | `django` — ~40 МБ, и это без DRF. | FastAPI. На стенде быстрее ставится, меньше шанс «а вдруг pip подвиснет». |
| **Где Django выиграл бы** | — | Если бы были: БД с миграциями, админка для оператора цеха, auth/многопользовательский режим, html-шаблоны с формами. | Ничего из этого в ТЗ нет (стадия 2 → «Что НЕ делаем сознательно»: «Мультиплеер, лидерборд на сервере, авторизация»). |

Итог: Django — это «фреймворк для CRUD-приложения с БД, юзерами и админкой». У нас нет ни CRUD, ни БД, ни юзеров. Это single-player симулятор с in-memory state, где ядро — быстрый async-endpoint к Claude и отдача JSON клиенту. FastAPI здесь не просто «модный», он буквально **минимальный достаточный** слой над нашим Python-кодом, тогда как Django заставил бы сначала отключить половину себя (через пустые `settings.py`, отключение middleware, неиспользуемую `db.sqlite3`), чтобы он не мешал.

> 1. **SQLite-snapshot, опционально, только для «сохранить партию → загрузить позже».** На hot-path всё равно in-memory `app.state`, БД пишется только по кнопке «сохранить» и читается при старте. > 2. **SQLite + SQLAlchemy 2.x (async) для истории партий/advice_log.** Подтягивает `sqlalchemy`, `aiosqlite` в requirements.txt. 4–6 часов буфера, может съесть время, которое нужно на копилот.
> 3. **Django полностью ради admin-панели оператора.** Переписывание всего стека, 10+ часов. В 36-часовом окне не укладывается — не рекомендую.
