Материал для уровней игры «Весёлый плановик». ## Источники-основа

- RaytecAPS: дискретное + гибридное производство, процессы (бленд/реакторы/линии розлива), Гант, коррекция по событиям, KPI: customer service level / WIP / setup losses. См. [raytec.pro/solutions/raytecaps](https://www.raytec.pro/solutions/raytecaps/).
- AVM.APS (1С:ERP-интеграция): уровни планирования (объёмный → межцеховой → пооперационный), критерии — длительность, опоздания, стоимость. См. [avmsky.ru/aps](https://avmsky.ru/aps/).
- 1С:ERP «Планирование производства»: сценарий → тип плана → период → номенклатура + дата запуска/выпуска. См. [efsol.ru/manuals/production-planning-erp](https://efsol.ru/manuals/production-planning-erp/).

| Уровень | Файл | Станков | Типов изделий | Заказов | События | Смысл уровня |
|---|---|---|---|---|---|---|
| 1. Обучающий | `level1_metalworking_small.json` | 5 | 3 (shaft, flange, bracket) | 6 | нет | Игрок учит UI: Гант, перетаскивание, «пролистать смену». Всё в срок по дефолту. |
| 2. Базовый | `level2_metalworking_medium.json` | 6 | 3 | 8 | 1 поломка на смену | Появляются одиночные события, копилот полезен, но можно и руками. |
| 3. Разветвлённые маршруты | `level3_assembly_workshop.json` | 6 (токарка / фрезерка / сварка / сборка / ОТК / упаковка) | 3 | 10 | 2 события/смена (breakdown + rush_order) | Длинные маршруты (до 7 операций для сборки), зависимости между операциями одного заказа, дедлайны плотные. |
| 4. Процессное | `level4_paint_production.json` | 6 (2 блендера, 2 реактора, 2 линии розлива) | 3 (грунт + 2 цвета краски) | 10 | переналадка между цветами + `supply_delay` | Большие штрафы за переналадку между цветами; копилот подсказывает группировку по цвету. |

Принцип распределения: каждый следующий уровень добавляет ровно один новый класс сложности (больше ресурсов → события → маршруты → переналадки), чтобы игрок понимал, что именно стало труднее. Жёсткий потолок на все уровни — **≤6 станков, ≤3 типов изделий, ≤10 заказов**, то есть не больше, чем в исходном `src/data/sample_orders.json` для task3. Это сознательное ограничение: ТЗ требует минимум 5 станков и 3 изделия, а больше — уже риск, что игрок утонет в деталях и не увидит сам факт планирования.

Уровень 5 («хардкор», 10 станков / 6 изделий / 25 заказов, все 6 типов событий) убран из линейки по этому же правилу. Его demo-сценарий (поломка фрезерного → копилот → жюри видит Ai2B) полностью укладывается в уровень 3 или 4 — достаточно вручную инжектировать поломку через `/api/event/random`, сцена читается так же.

## Формат файла

Единый JSON, совместимый со структурой dataclass'ов из `spec.md` (Machine / Order / Operation). Загружается в `src/main.py` через `load_plan(path)` (TODO — реализация после MVP-каркаса).

```jsonc
{
  "level": 1,
  "title_ru": "Небольшой металлообрабатывающий цех",
  "machines": [
    {"id": "M1", "type": "lathe",  "name_ru": "Токарный-1",  "setup_min": 5},
    {"id": "M2", "type": "mill",   "name_ru": "Фрезерный-1", "setup_min": 10}
  ],
  "products": {
    "shaft":  [{"machine_type": "lathe", "duration_min": 20}, {"machine_type": "qc", "duration_min": 5}],
    "flange": [{"machine_type": "mill",  "duration_min": 15}, {"machine_type": "qc", "duration_min": 5}]
  },
  "orders": [
    {"id": "ORD-001", "product": "shaft",  "qty": 3, "deadline_min": 240, "priority": 2},
    {"id": "ORD-002", "product": "flange", "qty": 2, "deadline_min": 180, "priority": 1}
  ],
  "events_profile": {"breakdown_per_shift": 0, "rush_order_per_shift": 0}
}
```

Поля `setup_min` / `events_profile` — задел на уровни 2-4 (переналадки и события), MVP первого уровня их игнорирует.
